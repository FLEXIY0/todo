# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Running the app

No build step for the web app. Open `index.html` directly in a browser or serve with any static server:
```
npx serve .
```

## Android (Capacitor)

The app ships as an Android APK via Capacitor (`android/`, `capacitor.config.json`). `npm run build` copies the web files into `www/`; `npx cap sync android` syncs them into the Android project. CI (`.github/workflows/build-apk.yml`) builds a signed release APK: pushing a `v*` tag publishes it to a GitHub Release; a manual run uploads it as an artifact. Release signing uses the committed `android/release.keystore` (fine for a personal sideloaded app; don't reuse for anything sensitive).

The launcher icon (pixel-art leaf) is generated by `node scripts/generate-icon.js` â€” a dependency-free PNG encoder that writes `icon.png`, `assets/icon*.png` and all `android/.../mipmap-*` launcher images from a 16Ă—16 pixel grid. Edit the grid/palette there and re-run, never edit the PNGs by hand. `scripts/generate-banner.js` builds the animated README banner (`docs/banner.svg`) from the same grid.

The app version is derived in CI from the release tag (`vX.Y` â†’ versionName `X.Y`, versionCode = run number); bump the defaults in `android/app/build.gradle` and `APP_VERSION` in `ui.js` when cutting a release.

## Architecture

Pure vanilla JS/CSS â€” no frameworks, no bundler, no dependencies.

**File roles:**
- `index.html` â€” markup only; no logic. Script order matters: `vendor/peerjs.min.js` â†’ `ui.js` â†’ `app.js` â†’ `sync.js`
- `ui.js` â€” all UI chrome: drawer, bottom sheet, dialog, theme switching, swipe gestures (drawer + space flip), long-press on empty space
- `app.js` â€” state, render loop, spaces, history journal, page-flip engine, task/category actions, animations
- `sync.js` â€” P2P sync of the shared space over WebRTC (vendored PeerJS; the public broker only introduces peers)
- `styles.css` â€” all styles including CSS keyframe animations

**State model (v2/v3):**
`state` in `app.js` holds `{ theme, spaces[], settings, history[], sync }`. Each space has `{ id, name, categories[], tree?, tabDot? }`; the shared one instead has `boards: { todo: [], wish: [] }` (two independent synced boards) with `mode` selecting the visible board â€” always access category lists via `spCats(space)`/`cats()`. Each category has `{ id, name, tasks[] }`; each task `{ id, text, done, mt }` plus optional `subtasks[]` (one level deep). `mt` is a last-modified timestamp used for sync merges. `loadState()` migrates v1 `{ categories }` and v2 shared-space shapes. Persisted to localStorage (`todo_state`). Subtasks render as stripes or a tree (`treeOn(space)`; per-space `tree` flag, shared follows its board).

**Spaces / page flip:**
`spaceIndex` indexes into `visSpaces()` (spaces filtered by `settings.wishlistOn/sharedOn`; the shared space is always last; `cats()` = current space's categories). Horizontal swipes drive a finger-controlled page-curl (`flipDragStart/Move/End` peeling a `.flip-page` layer from the corner); `flipTo(dir, mutate)` is the programmatic variant for the subtask/history/settings screens, and `flipBackDragStart/End` is the finger-driven "back" peel out of those nested screens (rightward swipe). Tabs in `#spaceTabs` jump via `flipToSpace()`. The single horizontal-gesture handler in `ui.js` routes a swipe by context: a rightward swipe opens the drawer only on the first space (full-screen, not just the edge), peels back out of a nested screen, or otherwise flips between spaces. In-progress drags (`backDrag`/`pageDrag`/`swActive`) are continued before any new gesture is re-decided. `settleFlip()` instantly finishes an in-flight peel tween so a fast follow-up swipe isn't dropped during the ~420ms animation.

**History journal:**
Every mutation calls `logH(sign, text, data?)` â€” entries with `data` are restorable via `restoreH()` (deleted/cleared items, text edits, wiped spaces). Retention is `settings.historyLimit` (50/200/1000/0=all), cycled from a chip on the history screen.

**Shared space sync:**
`sync.js` links devices through a "room" (`state.sync.room = { id, key }`; invite code = `id.key`). Many devices can share one room. Two channels: (1) direct P2P via PeerJS with peer ids derived from the room id (`st-<room>-a`/`-b` â€” deterministic rendezvous for the first two devices); (2) a minimal built-in MQTT-over-WebSocket client (QoS 0) publishing the AES-GCM-encrypted snapshot as a *retained* message to public brokers (`DEFAULT_BROKERS`, failover + backoff) â€” this is the multi-device bus / async mailbox, so devices need not be online together. `state.sync.brokers` (set via Sync â†’ "Use a custom server") overrides the brokers with your own wss/443 broker â€” the fix for carriers that block the public broker ports (some RU mobile networks); `state.sync.server` overrides the PeerJS server.

`saveState()` â†’ `maybeSync()` debounces sends; `mergeBoards()` does union + last-write-wins per board with tombstones (`state.sync.tombs`). Correctness rests on two things: **`mt` is a hybrid logical clock** (`nextMt()` in `app.js`) that never goes backwards relative to anything merged (`mergeBoards` advances `state.sync.clock` past the max it sees), so last-write-wins stays consistent even when phones' wall clocks disagree (the "done shows on one device only" bug); and **ids are globally unique** (`uid()` = device tag + time + counter), so concurrent adds never collide into phantom duplicates. A periodic `sendSnap(true)` every 11s heals dropped QoS-0 publishes. Echo loops prevented by payload dedup + own-device id. Every sync event is journaled with the 'â‡„' sign.

**Hardware back:**
A sentinel `history.pushState` entry (`armBack()`/`closeTopLayer()` in `ui.js`) makes the system back gesture close the topmost layer (dialog â†’ sheet â†’ drawer â†’ nested screen â†’ return to first space) instead of exiting; at the root the app exits normally.

**Subtasks:**
A task with subtasks cannot be toggled manually â€” `syncParentDone()` keeps its `done` derived from its subtasks (done â‡” all subtasks done). Tapping such a task opens the nested subtask screen (`subtaskView` in `app.js`, rendered by `renderSubtasks()`); on the main screen it shows one `.sub-bar` stripe per subtask under the text. The long-press sheet has a "Subtasks" item to enter the nested screen for any task.

**Render pattern:**
`render()` fully rebuilds `#categoriesContainer` innerHTML on every call. It is intentionally NOT called during task toggle animations â€” instead `toggleTask()` and `clearCompletedTasks()` do direct DOM class manipulation on elements identified by `data-id` / `data-cat-id` attributes to avoid interrupting CSS animations.

**Animation state:**
- `strikeForwardSet` / `strikeReverseSet` â€” Sets of task IDs currently animating strikethrough
- `animTimers` â€” Map of taskId â†’ setTimeout handle; used to cancel in-flight timers before re-toggling
- CSS animations (`strikeForward`, `strikeReverse`, `taskSlideOut`) are driven by adding/removing classes directly on DOM elements, not via re-render

**Long-press system:**
`setupLongPress(el, cb)` in `app.js` handles both touch and mouse with a 480ms timer and a 9px movement threshold. There is a separate inline long-press for empty space in `ui.js` using the same pattern.

**Category reorder:**
Category headers use `setupCategoryReorder()` (not `setupLongPress`): a long-press lifts the category (`.drag-lift`), dragging slides neighbours via transforms (`.drag-shift`) and reorders `state.categories` on drop; releasing without movement opens the category sheet instead.

**History / Settings / Themes / About:**
`historyView`, `settingsView` and `themesView` flags switch `render()` to the diff-styled journal screen, the settings screen (space options + a Fonts section) and the theme picker. All three are nested screens reached from the drawer, closed by back gesture/swipe, and registered in `closeTopLayer`/`nestedView`/`flipBackDragStart`. "About" is a plain sheet (`openAbout()` in `ui.js`, version in `APP_VERSION`).

**Theme & fonts:**
Four themes â€” `classic` (default amber/dark-brown), `oled` (pure black/white), `anthropic` (light ivory/terracotta) and `anthropic-dark` (warm `#2C2C2A` dark with antique-terracotta accent), listed in `THEMES` in `app.js` and rendered by `renderThemes` (the drawer has one "Themes" item; adding a theme needs a `THEMES` entry, a `.theme-dot.<id>` swatch and a CSS var block). `applyTheme()` toggles `body.theme-*`. Fonts are two `settings` keys (`fontSize` s/m/l, `fontFamily` system/mono/serif) applied by `applyDisplay()` via the `--font-scale` and `--app-font` CSS vars; readable text uses `calc(<px> * var(--font-scale))`. Category frame thickness is the `--cat-bw` var.

**Tap routing:**
`onTaskTap` (app.js) makes a single tap toggle a plain task instantly (history log deferred past the double-tap window so a quick second tap can cancel it) or open subtasks for a task that has them; a double tap always opens the subtask screen. `setupTripleTap(el, cb)` fires on the 3rd quick tap â€” used on category headers (clear that category's completed) and empty space (clear the space's completed, wired in `ui.js`).

**Bottom sheet / dialog:**
`openSheet(label, items)` and `openDialog(title, value, cb, isTask)` in `ui.js` are the only two modal primitives. All menus and confirmations go through these.
